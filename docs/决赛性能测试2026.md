# 2026 性能测试

## 题目目标

本题目要求参赛队伍在系统运行正确的基础上，对查询执行、存取管理、事务处理进行优化，使系统在 TPC-C 负载下获得更高吞吐率。本题目采用在快照隔离基础上进行 TPC-C 正确性测试。

性能测试关注两个方面：

1. 正确性：系统必须先通过功能测试、数据加载检查、压测后一致性检查和崩溃恢复检查。
2. 性能：系统在固定时间窗口内成功提交的 `NewOrder` 事务吞吐量越高，排名越靠前。

若任一正确性检查失败，则性能测试不参与排名，得分为 0。若全部检查通过，则性能测试得分为 100，并根据 tpmC 参与排行榜排名。

## Load 指令

在完成性能测试之前，参赛队伍需要完成 `load` 指令，把表的数据插入到系统中。

`load` 指令格式如下：

```sql
load file_name into table_name;
```

示例：

```sql
load ../../src/test/performance_test/table_data/warehouse.csv into warehouse;
```

其中，`file_name` 为数据文件相对于数据库文件夹的相对路径。数据文件在测试时存放于 `src/test/performance_test/table_data` 文件夹下。在测试脚本中，通过 `./bin/rmdb` 启动服务器，因此数据库文件夹位于 `build` 文件夹下。

`load_data` 指令不需要包含建表操作。

## 输出文件开关

为了减少将结果写入文件带来的时间开销，参赛队伍需要支持如下命令：

```sql
set output_file off
```

接收到该命令后，系统应停止向 `output.txt` 中写入输出结果。在未接收到该命令时，默认需要开启向 `output.txt` 中写入结果的功能。

注意：`set output_file off` 命令没有分号。

## 测试表与数据规模

测试程序中包含如下九张表：

- `warehouse` 表：记录仓库相关信息，表中有 `W` 条记录。
- `item` 表：记录商品信息，表中有 10 万条记录。
- `stock` 表：记录每个仓库的商品库存数据，表中有 `W * 10 万` 条记录。
- `district` 表：记录每个仓库提供服务的地区，表中有 `W * 10` 条记录。
- `customer` 表：记录每个地区的客户信息，表中有 `W * 10 * 3000` 条记录。
- `history` 表：记录客户的交易历史，表中有 `W * 10 * 3000` 条记录。
- `order` 表：记录每个地区的订单数据，表中有 `W * 10 * 3000` 条记录。
- `new_orders` 表：记录新订单信息，`order` 表中每个地区的最后 900 条订单被添加到 `new_orders` 表中。
- `order_line` 表：每个订单随机生成 5-15 条 `order_line` 记录。

其中，`W <= 50`。

在测试程序中，会对每张表的主键建立索引。测试事务中包含主键字段上的单点查询、范围查询和非主键字段上的单点查询、范围查询。

## 性能测试流程

性能测试采用固定时间窗口方式运行：

- 每轮先进行 30 秒预热，再进行 360 秒正式测量。
- 共重复 3 轮。
- 最终成绩以 3 轮正式测量阶段中成功提交的 `NewOrder` 事务吞吐量中位数作为性能指标，即 tpmC，单位为每分钟成功提交的新订单事务数。

事务占比如下：

- `10/23` `new_order`
- `10/23` `payment`
- `1/23` `order-status`
- `1/23` `delivery`
- `1/23` `stock-level`

在参与性能排名前，系统必须通过以下正确性检查：

- 功能测试全部通过。
- 数据加载结果与标准输出一致。
- 压测结束后，数据库表数据满足一致性检查。
- 模拟 `kill -9` 崩溃后重启，已提交事务的数据必须能够正确恢复。

数据加载完成后，测试程序会使用 `select count(*)` 检查各表记录数量，并进行数据一致性检验。压测结束后的数据一致性检验规则参考大赛仓库 `db2025` 中 `rmdb/数据一致性检验规则.pdf`。

评测机内存为 8GB，CPU 为 8 核。性能评测时，同时有 16 个客户端向服务端发起连接。测试程序最大运行时间为 2400 秒。

## 测试事务与 SQL 示例

测试程序包含五种事务：

- `NewOrder`：新订单的生成。
- `Payment`：订单付款。
- `OrderStatus`：最近订单查询。
- `Delivery`：配送。
- `StockLevel`：库存缺货状态分析。

以下 SQL 使用 `:name` 表示由测试程序填入的参数。

说明：测试表规模说明沿用 2026 初赛文档中的 `order` 表名称；以下 SQL 示例保留原事务示例中的 `orders` 表名，用于说明事务访问形态。

### NewOrder：读写事务

`NewOrder` 用于生成一笔新的订单交易。

```sql
select c_discount, c_last, c_credit, w_tax
from customer, warehouse
where w_id = :w_id
  and c_w_id = w_id
  and c_d_id = :d_id
  and c_id = :c_id;

select d_next_o_id, d_tax
from district
where d_id = :d_id
  and d_w_id = :d_w_id;

update district
set d_next_o_id = :d_next_o_id
where d_id = :d_id
  and d_w_id = :w_id;

insert into orders
values (:o_id, :o_d_id, :o_w_id, :o_c_id, :o_entry_d, :o_carrier_id,
        :o_ol_cnt, :o_all_local);

insert into new_orders
values (:o_id, :d_id, :w_id);
```

每条订单会包含多条 `order_line`。对每个订单项，测试程序会查询商品和库存，更新库存，并插入对应订单明细：

```sql
select i_price, i_name, i_data
from item
where i_id = :ol_i_id;

select s_quantity, s_data, s_dist_01, s_dist_02, s_dist_03, s_dist_04, s_dist_05,
       s_dist_06, s_dist_07, s_dist_08, s_dist_09, s_dist_10
from stock
where s_i_id = :ol_i_id
  and s_w_id = :ol_supply_w_id;

update stock
set s_quantity = :s_quantity
where s_i_id = :ol_i_id
  and s_w_id = :ol_supply_w_id;

insert into order_line
values (:ol_o_id, :ol_d_id, :ol_w_id, :ol_number, :ol_i_id,
        :ol_supply_w_id, :ol_delivery_d, :ol_quantity, :ol_amount, :ol_dist_info);
```

### Payment：读写事务

`Payment` 用于更新客户账户余额，并写入付款历史。

```sql
update warehouse
set w_ytd = w_ytd + :h_amount
where w_id = :w_id;

select w_street_1, w_street_2, w_city, w_state, w_zip, w_name
from warehouse
where w_id = :w_id;

update district
set d_ytd = d_ytd + :h_amount
where d_w_id = :w_id
  and d_id = :d_id;

select d_street_1, d_street_2, d_city, d_state, d_zip, d_name
from district
where d_w_id = :w_id
  and d_id = :d_id;

select c_first, c_middle, c_last, c_street_1, c_street_2, c_city, c_state, c_zip,
       c_phone, c_credit, c_credit_lim, c_discount, c_balance, c_since
from customer
where c_w_id = :w_id
  and c_d_id = :d_id
  and c_id = :c_id;

update customer
set c_balance = :c_balance
where c_w_id = :w_id
  and c_d_id = :d_id
  and c_id = :c_id;

insert into history
values (:h_c_id, :h_c_d_id, :h_c_w_id, :h_d_id, :h_w_id, :h_date,
        :h_amount, :h_data);
```

### Delivery：读写事务

`Delivery` 用于配送新订单，更新订单、订单明细和客户余额。

```sql
select min(no_o_id) as min_o_id
from new_orders
where no_d_id = :d_id
  and no_w_id = :w_id;

delete from new_orders
where no_o_id = :o_id
  and no_d_id = :d_id
  and no_w_id = :w_id;

select o_c_id
from orders
where o_id = :no_o_id
  and o_d_id = :d_id
  and o_w_id = :w_id;

update orders
set o_carrier_id = :o_carrier_id
where o_id = :no_o_id
  and o_d_id = :d_id
  and o_w_id = :w_id;

update order_line
set ol_delivery_d = :datetime
where ol_o_id = :no_o_id
  and ol_d_id = :d_id
  and ol_w_id = :w_id;

select sum(ol_amount) as sum_amount
from order_line
where ol_o_id = :no_o_id
  and ol_d_id = :d_id;

update customer
set c_balance = :c_balance,
    c_delivery_cnt = c_delivery_cnt + 1
where c_id = :c_id
  and c_d_id = :d_id
  and c_w_id = :w_id;
```

### OrderStatus：只读事务

`OrderStatus` 用于查询客户最近的交易状态。

60% 情况下，测试程序会先按客户姓氏查询客户数量，再按姓氏查询并排序客户信息：

```sql
select count(c_id) as count_c_id
from customer
where c_w_id = :c_w_id
  and c_d_id = :c_d_id
  and c_last = :c_last;

select c_balance, c_first, c_middle, c_last
from customer
where c_w_id = :c_w_id
  and c_d_id = :c_d_id
  and c_last = :c_last
order by c_first;
```

40% 情况下，测试程序会按客户 ID 查询客户信息：

```sql
select c_balance, c_first, c_middle, c_last
from customer
where c_w_id = :c_w_id
  and c_d_id = :c_d_id
  and c_id = :c_id;
```

随后查询该客户的订单和订单明细：

```sql
select o_id, o_entry_d, o_carrier_id
from orders
where o_w_id = :c_w_id
  and o_d_id = :c_d_id
  and o_c_id = :c_id
  and o_id = :o_id;

select ol_i_id, ol_supply_w_id, ol_quantity, ol_amount, ol_delivery_d
from order_line
where ol_w_id = :c_w_id
  and ol_d_id = :c_d_id
  and ol_o_id = :o_id;
```

### StockLevel：只读事务

`StockLevel` 用于查询仓库库存不足情况。

```sql
select d_next_o_id
from district
where d_id = :d_id
  and d_w_id = :w_id;

select ol_i_id
from order_line
where ol_w_id = :w_id
  and ol_d_id = :d_id
  and ol_o_id < :d_next_o_id
  and ol_o_id >= (:d_next_o_id - 20);

select count(*) as count_stock
from stock
where s_w_id = :w_id
  and s_i_id = :ol_i_id
  and s_quantity < :level;
```

## 优化思路

你可以根据大赛提供的优化思路对系统进行性能优化，也可以在系统中加入其他优化。优化思路如下：

- 完善系统的查询优化功能，生成更优的查询计划。
- 增加新的算子。
- 提高索引的并发度。
- 在日志管理器中增加线程池，实现日志的异步刷盘。

## 常见失败信息

如果第一阶段 `load_data` 超时，系统会输出：

```text
In performance test, server timeout in load_data phase
```

如果第一阶段或第二阶段事务运行过程中发生代码崩溃，系统会输出：

```text
In performance test, server stops running when processing tpcc transactions in the load_data/tpcc phase.
```

如果成功执行第二阶段事务，但一致性检验阶段发生代码崩溃，系统会输出：

```text
In performance test, server stops running when checking consistency.
```

并打印一致性检验相关信息。

如果成功执行第二阶段事务，但未通过一致性检验且未发生代码崩溃，系统会输出：

```text
Consistency check for xxxx failed
```
