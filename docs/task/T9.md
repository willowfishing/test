# 9. 2026题目九：可配置的快照隔离与可串行化隔离级别

**前置题目**：查询执行、事务控制语句

**题目种类**：基础功能

**推荐知识点**：事务可见性、隔离级别、快照隔离、可串行化、多版本并发控制、Serializable Snapshot Isolation、写写冲突、读写反依赖

## 题目描述

事务隔离级别决定了并发事务之间的可见性和冲突处理方式。不同隔离级别下，同一组交错执行的 SQL 语句可能产生不同的提交、回滚和查询结果。

本题聚焦于在 RMDB 中实现可配置的事务隔离级别：系统需要支持 `SNAPSHOT ISOLATION` 和 `SERIALIZABLE`，并能够通过 SQL 语句在运行时显式设置当前会话使用的隔离级别。测试只覆盖以下两种隔离级别：

1. `SNAPSHOT ISOLATION`
2. `SERIALIZABLE`

下文中，`SI` 表示 `SNAPSHOT ISOLATION`，`SER` 表示 `SERIALIZABLE`。

本题采用如下隔离级别模型：

1. `SI` 需要实现为基于 MVCC 的事务级快照隔离。
2. `SER` 需要实现为基于 MVCC 快照读的 Serializable Snapshot Isolation 风格判分模型，下文简称 `SSI`。

判分以本文定义的 MVCC 与 SSI 风格语义为准，检查 SQL 输出、事务回滚行为和最终数据库状态。

## 语法要求

系统需要支持以下语句：

```sql
SET TRANSACTION ISOLATION LEVEL SNAPSHOT ISOLATION;
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
```

其中：

1. `SET TRANSACTION ISOLATION LEVEL ...;` 用于显式设置当前会话后续新开启事务的隔离级别。
2. 除语法测试外，同一语义测试点内只会使用一种隔离级别，不会混合执行 `SNAPSHOT ISOLATION` 与 `SERIALIZABLE`，但不同语义测试点采用的隔离级别可能不同。
3. 每个会话中 `SET TRANSACTION ISOLATION LEVEL ...;` 只在会话开启后执行一次，不会出现事务`BEGIN`后再重新设置隔离级别的情况。
4. 显式事务中只包含增删改查四类语句，不包含 DDL 语句。

## 功能要求

参赛队伍需要实现以下能力：

1. 按当前事务隔离级别提供正确的事务可见性。
   - 写操作需要保留旧版本，或保留恢复旧版本所需的信息 (如`Undo Log`)，不能破坏事务按 MVCC 可见性规则应当看到的数据。
   - 已提交事务产生的写入结果，需要在满足隔离级别语义时对其他事务可见。
   - 事务应始终能读到自己已经写入但尚未提交的数据。
2. 支持显式事务的开始、提交与回滚。
   - 事务开始时需要记录本事务使用的隔离级别和快照标识；若为 `SER` 事务，还需要初始化读集合、写集合和依赖关系记录。
   - 事务提交后，依据隔离级别定义判断其他事务对该提交事务写入结果的可见性。
   - 事务回滚后，其在回滚前执行的 `INSERT`、`UPDATE`、`DELETE` 应等价于从未发生，且对其他事务不可见。
3. 在 `SI` 和 `SER` 下均需检测同一记录上的写写冲突。
   - 若事务 A 尝试更新或删除一条记录，而该记录的最新可写状态已经被另一个未提交事务 B 修改，则该写语句需要返回 `abort`，并回滚事务 A。
   - 若事务 A 基于旧快照尝试更新或删除一条记录，而该记录已经被其他事务在 A 开始之后提交修改，则该写语句需要返回 `abort`，并回滚事务 A。
4. 支持 `SI` 与 `SER` 两种隔离级别的显式配置。
   - 本题约定，除语法测试外，同一语义测试点的多个会话仅采用同一种隔离级别，不涉及混合隔离级别的并发执行，且每个会话都会先设置会话内隔离级别再执行事务。
   - 系统需要在运行时根据 `SET TRANSACTION ISOLATION LEVEL ...;` 的配置选择后续事务的隔离级别。
5. 满足 `SI` 与 `SER` 两种隔离级别的评测语义，其中 `SI` 按 MVCC 快照隔离级别评测，`SER` 按本文定义的 SSI 风格可串行化语义评测。

### 1. `SNAPSHOT ISOLATION`

本题中，`SNAPSHOT ISOLATION` 需要采用 MVCC 的事务级快照语义：

1. 每个事务在 `BEGIN;` 时获得一个开始时间戳或等价的快照标识，记为 `start_ts`。
2. 每个成功提交的事务获得一个单调递增的提交时间戳或等价的提交序，记为 `commit_ts`。
3. 对事务 T，某条记录的可见版本是满足 `commit_ts <= T.start_ts` 的最新已提交版本，再叠加 T 自己当前事务内尚未提交的写入。
4. 同一事务内对同一条件的多次读取，应基于同一个事务级快照返回结果；如果事务自身在两次读取之间写入了相关记录，则需要叠加自身写入后的结果。
5. 写操作不得原地覆盖或删除仍可能被活跃事务读取到的旧版本。可以通过版本链、Undo Log、删除标记或其他等价结构保留或重建旧版本。
6. 不允许脏读，不允许不可重复读。
7. 该级别需要检测同一逻辑记录上的写写冲突。两个并发事务不能都基于同一旧版本成功写入同一逻辑记录。

### 2. `SERIALIZABLE`

本题中，`SERIALIZABLE` 采用基于 MVCC 快照读的 Serializable Snapshot Isolation 风格判分模型 (`SSI`)。`SER` 继承 `SI` 的 MVCC 事务级快照读，并额外跟踪事务之间的读写反依赖，用于阻止不可串行化的危险结构。为保证判分结果确定，本题固定采用“当前语句所在事务”为危险结构的回滚对象。

本题对 `SER` 的要求为：

1. `SER` 事务的普通读取仍使用事务开始时的 MVCC 快照，并叠加本事务自身写入。
2. `SER` 事务执行 `SELECT` 时，需要保存本次查询的读信息：
   - 如果查询返回了某条记录，保存该记录的标识。
   - 如果查询带有条件或扫描范围，保存该查询条件或扫描范围；即使本次查询结果为空也需要保存。
   - 本题中的谓词读，指的就是事务保存过的查询条件或扫描范围。
3. `SER` 事务执行 `SELECT` 时，还需要检查其他相关 `SER` 事务已经产生、但当前事务快照不可见的写入。这里的写入包括其他活跃事务已经执行但尚未提交的写入，以及已提交但当前事务快照不可见的写入。若该写入会改变本次 `SELECT` 的结果，则形成一条读写反依赖，记为 current_reader ->rw writer。检查这些不可见写入仅用于建立 `rw` 反依赖，不改变 `SELECT` 的快照读结果。
4. `SER` 事务执行 `INSERT`、`DELETE`、`UPDATE` 时，需要将本次写操作影响的记录和其他相关 `SER` 事务保存的读信息比较：
   - `INSERT` 检查新插入的记录。
   - `DELETE` 检查被删除前的旧记录。
   - `UPDATE` 同时检查更新前的旧记录和更新后的新记录。
   - 如果被检查的记录是其他事务读过的记录，或满足其他事务保存过的查询条件或扫描范围，并且该读事务的快照中不包含本次写入，则形成一条读写反依赖，记为 reader ->rw current_writer。
5. 读写反依赖 (`rw` 反依赖) 的含义是：读事务按旧快照完成了读取，而写事务的写入会改变该读取在串行执行中的结果。单条 `rw` 反依赖本身不一定导致回滚。
6. SSI 危险结构指两个连续的 `rw` 反依赖：`Tin ->rw Tpivot` 和 `Tpivot ->rw Tout`。其中相关事务的执行区间存在重叠，并且满足 `Tin = Tout`，或 `Tout` 的提交顺序早于 `Tin`。
7. 若某事务因执行某条 `SELECT`、`INSERT`、`UPDATE` 或 `DELETE` 语句新增了 `rw` 反依赖，并且新增后形成 SSI 危险结构，则执行该语句的事务需立即中止并回滚，该语句向客户端返回 `abort`。
8. 所有 `rw` 反依赖应在 `SELECT`、`INSERT`、`UPDATE`、`DELETE` 执行时检测，且危险结构应在新增依赖的语句处处理。
9. 同一逻辑记录上的写写冲突按 `SI` 规则处理；也就是两个并发事务不能都成功修改同一条旧版本记录。
10. 已回滚或被系统中止的事务应视为从未发生。事务回滚后，其已经执行的 `INSERT`、`UPDATE`、`DELETE` 不应对其他事务可见，也不应影响最终数据库状态；该事务产生的写集合、读集合、谓词读记录以及 `rw` 反依赖关系应被清理，后续危险结构检测不应再受到该事务影响。

## 判分语义说明

为避免歧义，测试将按如下原则判断：

1. `SI` 需要保证 MVCC 事务级快照的一致性和写写冲突检测。
2. `SER` 需要在 `SI` 的基础上实现符合本文所定义的 SSI 风格语义，以及相应的 `rw` 反依赖跟踪和危险结构检测。
3. 除语法测试外，单个测试点内只测试同一隔离级别，单个测试点内的每个会话都会被设置到相同隔离级别。设置为 `SNAPSHOT ISOLATION` 的测试点按 `SI` 规则判分；设置为 `SERIALIZABLE` 的测试点按 `SER` 规则判分。
4. 未形成写写冲突或 SSI 危险结构的调度，按 MVCC 快照读结果和事务提交结果判分。
5. 写写冲突需要在冲突写语句返回 `abort`，并回滚执行该写语句的事务。`SELECT`、`INSERT`、`UPDATE` 或 `DELETE` 语句在执行时对 `rw` 反依赖进行检测并记录；若新增 `rw` 反依赖后形成 SSI 危险结构，则由新增该依赖的语句返回 `abort`，并回滚该语句所在事务。

## 实现方法提示

以下提示仅供参考：

1. 在会话状态中保存当前配置的隔离级别，并在 `BEGIN;` 时复制到事务上下文中。
2. 为事务分配开始时间戳、提交时间戳或等价的可见性标识。
3. 为记录维护版本链、撤销日志 (`Undo Log`) 或其他可回溯的历史版本结构，以支持事务级快照读和回滚。
4. 事务应能够读到自己当前事务内尚未提交的写入。
5. `SER` 可以在 `SI` 的版本可见性基础上增加记录读集合、谓词读集合、写集合和事务依赖图。
6. 记录读可按记录标识跟踪；范围读、全表扫描和空结果查询可保存查询条件或扫描范围，用于在后续写操作时判断新旧记录是否会改变该次查询结果。
7. `SER` 实现时，可以在读语句执行时检查当前事务快照不可见、但会改变本次查询结果的其他事务写入，建立 current_reader ->rw writer；在写语句执行时用其他事务的记录读和谓词读建立 reader ->rw current_writer。每次新增 `rw` 反依赖后，都需要检查是否形成危险结构，若某条语句在执行时新增了 `rw` 依赖并形成了SSI危险结构，则中止该语句，向客户端报告 abort 并回滚该语句所在事务。
8. 已回滚或被系统中止的事务应视为从未发生。事务回滚后，其已经执行的 `INSERT`、`UPDATE`、`DELETE` 不应对其他事务可见，也不应影响最终数据库状态；该事务产生的读集合、谓词读记录、写集合以及 `rw` 反依赖关系应被清理，后续可见性判断、冲突检测和危险结构检测不应再受到该事务影响。
9. 事务提交后，其提交版本仍属于数据库状态。实现可以释放事务运行期的临时状态，但对于仍可能被并发 `SER` 事务用于 `rw` 反依赖和危险结构检测的信息，例如必要的读写集合、提交顺序和依赖关系，应保留到不再有相关并发事务需要使用时再清理。

## 测试示例

下面给出若干参考示例帮助理解本题的判分方式。示例中的 SQL 按逻辑划分为 `T1`、`T2`、`T3` 三个事务，实际测试中可能由多个会话交错执行。

### 示例一：基本并发样例，同一记录上的写写冲突

该样例用于验证对同一条记录的并发写冲突处理。该样例在 `SI` 和 `SER` 下结果一致。

初始化 SQL：

```sql
create table account (id int, balance int);
insert into account values (1, 100);
```

事务 1：

```sql
t1s set transaction isolation level <LEVEL>;
t1a begin;
t1b update account set balance = 120 where id = 1;
t1c commit;
```

事务 2：

```sql
t2s set transaction isolation level <LEVEL>;
t2a begin;
t2b update account set balance = 90 where id = 1;
t2c commit;
```

事务 3：

```sql
t3a select * from account where id = 1;
```

调度顺序如下：

| 步骤 | `T1`                                                 | `T2`                                                | `T3`                                      |
| ---- | ---------------------------------------------------- | --------------------------------------------------- | ----------------------------------------- |
| 1    | `t1s set transaction isolation level <LEVEL>;`       |                                                     |                                           |
| 2    |                                                      | `t2s set transaction isolation level <LEVEL>;`      |                                           |
| 3    | `t1a begin;`                                         |                                                     |                                           |
| 4    | `t1b update account set balance = 120 where id = 1;` |                                                     |                                           |
| 5    |                                                      | `t2a begin;`                                        |                                           |
| 6    |                                                      | `t2b update account set balance = 90 where id = 1;` |                                           |
| 7    | `t1c commit;`                                        |                                                     |                                           |
| 8    |                                                      | `t2c commit;`                                       |                                           |
| 9    |                                                      |                                                     | `t3a select * from account where id = 1;` |

期望标准输出：

```text
abort
+------------------+------------------+
|               id |          balance |
+------------------+------------------+
|                1 |              120 |
+------------------+------------------+
Total record(s): 1
```

样例说明：

1. `T2` 不能在 `T1` 未结束时成功覆盖 `T1` 对同一记录的更新。
2. `T2` 应在冲突写语句 `t2b` 返回 `abort`，并回滚事务 2。
3. 若 `T2` 已经在 `t2b` 被回滚，则后续 `t2c` 不应使事务 2 的写入生效。
4. `T3` 的查询结果应为 `(1, 120)`。

### 示例二：事务级快照的一致性

该样例用于验证事务级快照语义。在 `SI` 与 `SER` 下，事务读取的都是其开始时的 MVCC 快照，再叠加本事务自身写入。

初始化 SQL：

```sql
create table counter_test (id int, val int);
insert into counter_test values (1, 100);
```

事务 1：

```sql
t1s set transaction isolation level <LEVEL>;
t1a begin;
t1b select * from counter_test where id = 1;
t1c select * from counter_test where id = 1;
t1d commit;
```

事务 2：

```sql
t2a begin;
t2b update counter_test set val = 200 where id = 1;
t2c commit;
```

调度顺序如下：

| 步骤 | `T1`                                           | `T2`                                                  |
| ---- | ---------------------------------------------- | ----------------------------------------------------- |
| 1    | `t1s set transaction isolation level <LEVEL>;` |                                                       |
| 2    | `t1a begin;`                                   |                                                       |
| 3    | `t1b select * from counter_test where id = 1;` |                                                       |
| 4    |                                                | `t2a begin;`                                          |
| 5    |                                                | `t2b update counter_test set val = 200 where id = 1;` |
| 6    |                                                | `t2c commit;`                                         |
| 7    | `t1c select * from counter_test where id = 1;` |                                                       |
| 8    | `t1d commit;`                                  |                                                       |

当 `<LEVEL> = SNAPSHOT ISOLATION` 时，期望标准输出为：

```text
+------------------+------------------+
|               id |              val |
+------------------+------------------+
|                1 |              100 |
+------------------+------------------+
Total record(s): 1
+------------------+------------------+
|               id |              val |
+------------------+------------------+
|                1 |              100 |
+------------------+------------------+
Total record(s): 1
```

样例说明：

1. 两种隔离级别都不允许脏读。
2. 在同一事务中，即使并发事务已经提交新的写入结果，`T1` 的第二次查询仍应保持与第一次查询一致。
3. 该样例主要验证 `SI` 必须满足事务级快照的一致性。

### 示例三：写偏序场景

写偏序是用于区分 `SNAPSHOT ISOLATION` 与 `SERIALIZABLE` 隔离级别的关键场景之一。

场景描述：

表 `duty` 记录两位值班医生是否在岗，约束为“任意时刻至少有一位医生在岗”。初始时两位医生都在岗。两个事务分别读取“另一位医生仍在岗”的事实后，都将自己改为离岗。该调度会形成相反方向的读写反依赖，是典型写偏序异常。

初始化 SQL：

```sql
create table duty (doctor_id int, on_call int);
insert into duty values (1, 1);
insert into duty values (2, 1);
```

事务 1：

```sql
t1s set transaction isolation level <LEVEL>;
t1a begin;
t1b select * from duty where doctor_id = 2;
t1c update duty set on_call = 0 where doctor_id = 1;
t1d commit;
```

事务 2：

```sql
t2s set transaction isolation level <LEVEL>;
t2a begin;
t2b select * from duty where doctor_id = 1;
t2c update duty set on_call = 0 where doctor_id = 2;
t2d commit;
```

事务 3：

```sql
t3a select * from duty;
```

调度顺序如下：

| 步骤 | `T1`                                                   | `T2`                                                   | `T3`                      |
| ---- | ------------------------------------------------------ | ------------------------------------------------------ | ------------------------- |
| 1    | `t1s set transaction isolation level <LEVEL>;`         |                                                        |                           |
| 2    |                                                        | `t2s set transaction isolation level <LEVEL>;`         |                           |
| 3    | `t1a begin;`                                           |                                                        |                           |
| 4    |                                                        | `t2a begin;`                                           |                           |
| 5    | `t1b select * from duty where doctor_id = 2;`          |                                                        |                           |
| 6    |                                                        | `t2b select * from duty where doctor_id = 1;`          |                           |
| 7    | `t1c update duty set on_call = 0 where doctor_id = 1;` |                                                        |                           |
| 8    |                                                        | `t2c update duty set on_call = 0 where doctor_id = 2;` |                           |
| 9    | `t1d commit;`                                          |                                                        |                           |
| 10   |                                                        | `t2d commit;`                                          |                           |
| 11   |                                                        |                                                        | `t3a select * from duty;` |

当 `<LEVEL> = SNAPSHOT ISOLATION` 时，期望标准输出为：

```text
+------------------+------------------+
|        doctor_id |          on_call |
+------------------+------------------+
|                2 |                1 |
+------------------+------------------+
Total record(s): 1
+------------------+------------------+
|        doctor_id |          on_call |
+------------------+------------------+
|                1 |                1 |
+------------------+------------------+
Total record(s): 1
+------------------+------------------+
|        doctor_id |          on_call |
+------------------+------------------+
|                1 |                0 |
|                2 |                0 |
+------------------+------------------+
Total record(s): 2
```

此时 `T1` 和 `T2` 均可成功提交。`T1` 与 `T2` 读取的是各自事务开始时的快照，因此都能看到“另一位医生仍在岗”；由于两个事务更新的是不同记录，不发生同一记录上的写写冲突，最终可能出现两人都离岗的写偏序结果。

当 `<LEVEL> = SERIALIZABLE` 时，期望标准输出为：

```text
+------------------+------------------+
|        doctor_id |          on_call |
+------------------+------------------+
|                2 |                1 |
+------------------+------------------+
Total record(s): 1
+------------------+------------------+
|        doctor_id |          on_call |
+------------------+------------------+
|                1 |                1 |
+------------------+------------------+
Total record(s): 1
abort
+------------------+------------------+
|        doctor_id |          on_call |
+------------------+------------------+
|                1 |                0 |
|                2 |                1 |
+------------------+------------------+
Total record(s): 2
```

在 `SER` 下，`t1c` 先建立 `T2 ->rw T1`，尚未形成危险结构；`t2c` 再建立 `T1 ->rw T2`，此时形成 SSI 危险结构。根据本题固定的当前语句回滚规则，`t2c` 需要返回 `abort`，`T2` 被回滚，最终只有 `T1` 的更新生效。

## 测试点说明

测试点包括但不限于以下内容：

| 测试点                            | 详细说明                                                     |
| --------------------------------- | ------------------------------------------------------------ |
| `si/InsertTest`                   | 在 `SI` 下验证插入记录的事务内可见性、并发事务快照可见性以及提交后的最终查询结果 |
| `si/DirtyReadTest`                | 在 `SI` 下验证未提交更新不会被其他事务读到，同时事务能够读到自己的未提交写入 |
| `si/ReadWriteConflictDeleteTest`  | 在 `SI` 下验证读后并发删除、再删除同一记录时的快照可见性和删除冲突处理 |
| `si/WriteWriteConflictUpdateTest` | 在 `SI` 下验证两个事务并发更新同一记录时的写写冲突处理，最终只能保留一个有效写入结果 |
| `ser/UnrepeatableReadTest`        | 在 `SER` 下验证重复读取同一记录时的快照读结果和 SSI 依赖判断 |
| `ser/PhantomReadTest`             | 在 `SER` 下验证范围读、空结果读与并发写入之间的谓词读依赖判断 |
| `ser/WriteSkewTest`               | 在 `SER` 下执行写偏序调度，验证系统能够回滚导致 SSI 危险结构成立的事务并维持全局约束 |

## 测试输出要求

1. 对于写写冲突，冲突写语句需要向客户端返回 `abort\n`，并回滚执行该写语句的事务。
2. 对于 SSI 危险结构，新增 `rw` 反依赖并导致危险结构成立的当前语句需要向客户端返回 `abort\n`，并回滚当前语句所在事务；当前语句可以是 `SELECT`、`INSERT`、`UPDATE` 或 `DELETE`。
3. 除 `select` 的标准输出以及必要的事务回滚信息之外，不应向客户端返回多余信息。
4. 服务端的输出仍应写入对应数据库目录下的 `output.txt` 文件中。
